# Visual Montage Pipeline 交接文档（2026-07-16）

## 1. 项目目标

项目用于批量生成画面向广告混剪。当前重点品类为 Beauty，目标是从多条
UGC 视频中识别高价值画面，生成低重复的 20 秒广告，并输出可继续编辑的剪映
草稿。

当前广告结构：

```text
0.0–0.8    视觉 Hook
0.8–7.2    第一段高光混剪
7.2–8.7    Rednote 产品开屏
8.7–12.8   产品录屏
12.8–17.5  第二段高光混剪
17.5–20.0  尾贴和 CTA
```

产品开屏与录屏共用一条连续口播。口播从产品开屏开始，可延伸到第二段高光，
但必须在尾贴前结束。

## 2. 今天完成的工作

### 2.1 分析失败不再阻断整批任务

旧逻辑中，只要任意一条素材的远程分析失败，`analyze_visual_batch.py` 就返回
退出码 1。即使其余素材已经产生足够候选，`batch-run` 仍会停止。

现在的行为：

- 单条素材失败会写入 `candidate-pool.json.failures`。
- 只要存在可用候选，分析阶段就正常结束并继续编排。
- 只有所有素材都没有生成候选时才终止。
- 最终 `result.json` 会包含 `analysis_failures` 和 `warnings`。

本次 15 条 Beauty 素材最终分析成功：

```text
videos: 15
candidates: 50
failures: 0
cache_hit: 13
cache_miss: 2
```

### 2.2 修复英文口播“两句话”校验

旧校验器只识别中文标点 `。！？`，英文口播虽然包含两个句号，仍被错误判定为
少于两句话。

现在同时支持：

```text
中文：。！？
英文：.!?
```

### 2.3 Marlin 超长视频分段策略

当前 Marlin 服务单次最大接受 120 秒视频。本批两条素材分别约 140.5 秒和
188.6 秒，旧流程会得到：

```text
marlin_failed_gemma_fallback
```

实际原因是：

```text
Input video duration exceeds maximum 120.0s for Marlin inference
```

现在路由为：

```text
< 30 秒
→ Gemma 完整代理视频分析

30–120 秒
→ Marlin 事件召回
→ Gemma 完整代理视频复核

> 120 秒
→ Marlin 分段事件召回
→ 分段时间戳还原为原视频绝对时间
→ 合并召回结果
→ Gemma 完整代理视频复核
```

Beauty 当前分段配置：

```yaml
marlin_recall:
  minimum_video_duration_seconds: 30.0
  maximum_video_duration_seconds: 120.0
  segment_duration_seconds: 115.0
  segment_overlap_seconds: 5.0
```

例如 188.6 秒视频会切为：

```text
0–115 秒
110–188.6 秒
```

重叠 5 秒用于降低事件被切点截断的风险。成功路由标识为：

```text
marlin_segmented_then_gemma
```

缓存已做针对性失效：

- 120 秒以内视频继续复用旧分析缓存。
- 超过 120 秒的视频会使用新分段配置生成新的缓存键。
- 不需要为了启用分段策略对全部素材使用 `--force-analysis`。

### 2.4 自适应口播与 BGM 音量

测试批次中口播平均音量约为 `-23.1 dB`，部分 BGM 平均音量达到 `-8 dB`。
旧流程只在口播区间固定降低 BGM `-7 dB`，因此口播仍然偏小。

现在每条成片都会：

1. 使用 FFmpeg 测量口播平均音量和峰值。
2. 测量所选 BGM 的平均音量和峰值。
3. 在不超过峰值上限的条件下计算口播增益。
4. 根据每首 BGM 的实际响度计算基础音量。
5. 根据口播输出响度和目标响度差计算口播期间的 BGM ducking。

Campaign 当前参数：

```yaml
audio:
  voice_target_mean_db: -16
  voice_peak_ceiling_db: -1
  bgm_target_mean_db: -20
  voiceover_margin_db: 12
  fade_out_seconds: 0.5
```

这些是目标值，不是直接写死的剪映音量。每个工程的实际 `volume` 和
`voiceover_ducking_db` 会按素材测量结果单独计算，并写入
`jianying-plan.json`。

### 2.5 BGM 时长与成片对齐

旧导出器假定所选 BGM 至少有 20 秒。测试中一首 BGM 实际只有 16.882 秒，
导出器仍尝试读取到 20 秒，导致 `beauty-full-v1-004` 剪映工程验证失败。

现在的规则：

- BGM 长于成片：只使用成片所需时长。
- BGM 等于成片：正常使用。
- BGM 短于成片：从头循环补足完整成片。
- 循环点会加入剪映音频切片边界，避免单个切片越过源素材长度。
- BGM 轨道目标时长始终等于成片时长。

导出结果会记录：

```json
{
  "source_duration_seconds": 16.882,
  "looped": true,
  "base_volume": 0.0,
  "ducking_volume": 0.0
}
```

上述音量仅为字段示例，实际值由自适应响度计算产生。

### 2.6 英文字幕保留单词空格

旧字幕逻辑为了处理中文，无条件删除了全部空白字符，导致：

```text
OpenRednoteforendlessbeautyinspiration
```

现在会根据文字类型选择切分方式：

- 中文按句号、逗号和短语切分。
- 英文按句子和单词切分。
- 英文完整保留单词空格。
- 过长英文句子约每 8 个词切为一段。
- 所有字幕段首尾仍与 TTS 总时长一致。
- 字幕继续作为剪映原生可编辑文字轨道输出。

字幕缓存版本已升级。下次生成计划时，旧的无空格字幕缓存不会继续使用。

### 2.7 英文封面标题

`profiles/categories/beauty.yaml` 中的 `batch_generation.cover_titles` 已改为英文。
英文标题会作为剪映原生可编辑文字输出，不会烘焙成唯一封面成品。

当前标题可正常显示，但仍建议：

- 优先使用 4–8 个英文单词。
- 避免过长标题占满安全区。
- 标题含冒号等 YAML 特殊字符时使用引号。
- 当前字体为 `douyin_sans_bold_zh`，包含常用英文字形。

当前列表中存在少量重复标题，后续可人工去重。

## 6. 验收重点

新一批生成后必须人工检查：

1. 五个剪映工程是否全部存在，`skipped_count` 是否为 0。
2. 口播是否明显高于口播区间内的 BGM，同时没有削波爆音。
3. 口播结束后 BGM 是否自然恢复。
4. BGM 是否完整覆盖 0–20 秒。
5. 短 BGM 循环点是否有明显跳变。
6. 英文字幕是否保留空格，字幕时间是否跟随口播。
7. 英文封面标题是否超出安全区或出现不理想换行。
8. 产品开屏是否紧邻产品录屏。
9. 封面使用 PNG Logo，首个高光使用动态 MOV Logo，后续使用 PNG Logo。
10. 五条视频之间是否存在明显重复镜头。

## 8. 成本与缓存注意事项

- 不要随意使用 `--force-analysis`，否则会重新调用 Gemma 和 Marlin。
- 不要随意使用 `--force-audio`，否则会重新分析所有素材音频。
- 修改 Gemma Prompt、Marlin Query 或关键分析配置会产生新缓存键。
- 修改封面标题、混音目标、Logo 或 Campaign 包装不应该触发视频理解重跑。
- `candidate-registry.sqlite` 同时保存候选、候选使用历史、视频分析缓存、BGM
  资产和 BGM 使用历史，不要在批次之间随意删除。
- 成功导出后候选和 BGM 状态为 `committed`；失败或 partial 应释放 reservation。

- Marlin 仅用于候选召回，不能直接把 Marlin 结果作为最终镜头。

- 候选不足时返回 partial，不使用弱镜头凑数。
- 保留产品开屏紧邻产品录屏。
- 口播必须保持一条连续音频，可拆字幕但不要拆成两条 TTS。
- 封面标题必须保持剪映可编辑。
- 修改代码后先在开发目录运行测试，再同步到实际运行目录。
- 区分旧 Run 的产物与新代码能力，不能用旧结果判断新修复是否生效。
